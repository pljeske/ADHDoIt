## Project overview

ADHDoIt is a multi-user todo list application designed with ADHD-friendly UX in mind. It features smart views, overdue overflow (todos past deadline are never lost but don't clutter the main list), date/time reminders via email and Web Push, categories, and flexible sorting. The backend is a stateless Go REST API; the frontend is a React SPA. Both run as containers via Docker/Podman Compose.

## Database schema

### Migrations

Use `golang-migrate` with numbered SQL files in `backend/internal/db/migrations/`. Run automatically at server startup via `migrate.Up()`.

#### `000001_init.up.sql`

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TYPE todo_status AS ENUM ('active', 'snoozed', 'done');

CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email       TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    name        TEXT NOT NULL,
    timezone    TEXT NOT NULL DEFAULT 'UTC',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE refresh_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE,
    expires_at  TIMESTAMPTZ NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON refresh_tokens(user_id);
CREATE INDEX ON refresh_tokens(expires_at);

CREATE TABLE categories (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT NOT NULL,
    color       TEXT NOT NULL DEFAULT '#6366f1',  -- hex color
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON categories(user_id);

CREATE TABLE todos (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    category_id  UUID REFERENCES categories(id) ON DELETE SET NULL,
    title        TEXT NOT NULL,
    description  TEXT,
    deadline     DATE NOT NULL,              -- defaults to today in user's timezone, set by API
    reminder_at  TIMESTAMPTZ,               -- null = no reminder
    priority     SMALLINT NOT NULL DEFAULT 0 CHECK (priority BETWEEN 0 AND 3),
    status       todo_status NOT NULL DEFAULT 'active',
    snooze_until DATE,                      -- set when status = 'snoozed'
    done_at      TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON todos(user_id, status, deadline);
CREATE INDEX ON todos(user_id, category_id);
CREATE INDEX ON todos(reminder_at) WHERE reminder_at IS NOT NULL AND status != 'done';

CREATE TABLE push_subscriptions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    endpoint    TEXT NOT NULL UNIQUE,
    p256dh      TEXT NOT NULL,
    auth        TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON push_subscriptions(user_id);
```

Also create the River job queue tables using `river/rivermigrate` — call `rivermigrate.New(driver).Up(ctx, nil)` at startup, after your own migrations run.

## sqlc setup

`backend/internal/db/sqlc.yaml`:

```yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "queries/"
    schema: "migrations/"
    gen:
      go:
        package: "db"
        out: "."
        sql_package: "pgx/v5"
        emit_result_struct_pointers: true
        emit_params_struct_pointers: true
```

Write all SQL queries in `backend/internal/db/queries/`. Run `sqlc generate` to regenerate. Commit generated files.

Key query files to create:

- `users.sql` — GetUserByEmail, GetUserByID, CreateUser, UpdateUserTimezone
- `todos.sql` — CreateTodo, GetTodo, ListTodos (parameterized by user/status/deadline filters), UpdateTodo, DeleteTodo, GetTodosWithDueReminder
- `categories.sql` — CreateCategory, ListCategories, GetCategory, UpdateCategory, DeleteCategory
- `refresh_tokens.sql` — CreateRefreshToken, GetRefreshToken, DeleteRefreshToken, DeleteExpiredRefreshTokens
- `push_subscriptions.sql` — UpsertPushSubscription, ListPushSubscriptionsByUser, DeletePushSubscription

For `ListTodos`, use a parameterized query with filters. Example approach — use sqlc's `sqlc.narg()` for nullable params or write separate named queries per view. Named queries per view is simpler and more readable:

```sql
-- name: ListTodosToday :many
SELECT * FROM todos
WHERE user_id = $1
  AND status = 'active'
  AND deadline = (CURRENT_DATE AT TIME ZONE $2)::date
ORDER BY priority DESC, deadline ASC, created_at ASC;

-- name: ListTodosUpcoming :many
SELECT * FROM todos
WHERE user_id = $1
  AND status = 'active'
  AND deadline > CURRENT_DATE AT TIME ZONE $2
ORDER BY deadline ASC, priority DESC, created_at ASC;

-- name: ListTodosOverdue :many
SELECT * FROM todos
WHERE user_id = $1
  AND status IN ('active', 'snoozed')
  AND deadline < CURRENT_DATE AT TIME ZONE $2
ORDER BY deadline ASC, priority DESC, created_at ASC;

-- name: ListTodosDone :many
SELECT * FROM todos
WHERE user_id = $1
  AND status = 'done'
ORDER BY done_at DESC;

-- name: ListTodosByCategory :many
SELECT * FROM todos
WHERE user_id = $1
  AND category_id = $2
  AND status != 'done'
ORDER BY deadline ASC, priority DESC, created_at ASC;
```

## API design

Base path: `/api/v1`

All authenticated endpoints require `Authorization: Bearer <access_token>`.

All request and response bodies are JSON.

### Auth

```
POST   /api/v1/auth/register
POST   /api/v1/auth/login
POST   /api/v1/auth/refresh
DELETE /api/v1/auth/logout
```

**`POST /auth/register`**
```json
// Request
{ "email": "user@example.com", "name": "Lucas", "password": "min8chars", "timezone": "Europe/Berlin" }

// Response 201
{ "user": { "id": "uuid", "email": "...", "name": "...", "timezone": "..." }, "access_token": "...", "refresh_token": "..." }
```

**`POST /auth/login`**
```json
// Request
{ "email": "user@example.com", "password": "..." }

// Response 200 — same shape as register
```

**`POST /auth/refresh`**
```json
// Request
{ "refresh_token": "..." }

// Response 200
{ "access_token": "...", "refresh_token": "..." }  // refresh token is rotated
```

**`DELETE /auth/logout`** — authenticated; deletes the refresh token from DB. Returns 204.

---

### Todos

```
GET    /api/v1/todos?view=today|upcoming|overdue|done|all&category_id=<uuid>&sort=deadline|priority|created_at&order=asc|desc
POST   /api/v1/todos
GET    /api/v1/todos/:id
PATCH  /api/v1/todos/:id
DELETE /api/v1/todos/:id

POST   /api/v1/todos/:id/snooze
POST   /api/v1/todos/:id/done
POST   /api/v1/todos/:id/reopen
```

**`GET /todos`** — returns array of todo objects filtered by `view`. Default view: `today`. All todos belong to the authenticated user.

**`POST /todos`**
```json
// Request — deadline and priority are optional
{
  "title": "Buy groceries",
  "description": "Milk, eggs",
  "category_id": "uuid-or-null",
  "deadline": "2025-04-10",      // optional; defaults to today in user's timezone
  "priority": 2,                 // optional; defaults to 0
  "reminder_at": "2025-04-10T09:00:00Z"  // optional
}

// Response 201
{ "id": "uuid", "title": "...", "deadline": "...", "status": "active", ... }
```

**`PATCH /todos/:id`** — partial update; any field from the create body is patchable. If `reminder_at` changes, cancel old River job and enqueue new one. If `reminder_at` is set to null, cancel existing job.

**`POST /todos/:id/snooze`**
```json
// Request
{ "snooze_until": "2025-04-15" }
// Sets status = 'snoozed', snooze_until = provided date
// Response 200 — updated todo
```

**`POST /todos/:id/done`** — sets `status = done`, `done_at = NOW()`. Returns 200.

**`POST /todos/:id/reopen`** — sets `status = active`, clears `done_at`, `snooze_until`. Returns 200.

**`DELETE /todos/:id`** — hard delete. Returns 204.

All errors follow:

HTTP status codes: 400 bad request, 401 unauthenticated, 403 forbidden (wrong user), 404 not found, 409 conflict (duplicate email), 422 validation error, 500 internal.

## Go server structure

### `cmd/server/main.go`

1. Load config from env
2. Connect to PostgreSQL via `pgxpool.New()`
3. Run migrations (golang-migrate, then River migrations)
4. Build sqlc `Queries` struct
5. Start River workers in background goroutine (reminder dispatch worker)
6. Build chi router with middleware
7. `http.ListenAndServe()`
8. Handle OS signals for graceful shutdown (cancel context, drain River, close pool)

### Middleware stack (in order)

```go
r.Use(middleware.RealIP)
r.Use(middleware.RequestID)
r.Use(slogMiddleware)          // structured request logging
r.Use(cors.Handler(cors.Options{...}))
r.Use(middleware.Recoverer)
r.Use(middleware.Timeout(30 * time.Second))
```

Auth middleware is applied per-route group, not globally.

### Handler pattern

Use a `Handler` struct per resource that holds `*db.Queries` and config:

```go
type TodoHandler struct {
    q      *db.Queries
    pool   *pgxpool.Pool   // needed for River client
    river  *river.Client[pgx.Tx]
    cfg    *config.Config
}
```

Decode request → validate with `validator` → call sqlc query → return JSON. No business logic outside handlers for the initial implementation. Inline helper `respondJSON(w, status, v)` and `respondError(w, status, msg, code)`.

### JWT implementation

- Access token: short-lived (15m), contains `user_id` and `email` in claims. Signed with HS256.
- Refresh token: random 32-byte value, stored as SHA-256 hash in `refresh_tokens` table, returned as hex string to client.
- Auth middleware: extract Bearer token → parse JWT → inject user ID into context via typed key.
- On refresh: verify token against DB hash, delete old token, issue new pair (rotation).

## River worker

Define one job type: `ReminderArgs`:

```go
type ReminderArgs struct {
    TodoID string `json:"todo_id"`
    UserID string `json:"user_id"`
}

func (ReminderArgs) Kind() string { return "reminder" }
```

Worker implementation:
1. Fetch the todo — if not found or status = done, skip silently
2. Fetch user (for email, name, timezone)
3. Fetch user's push subscriptions
4. Send email via SMTP (goroutine)
5. Send web push to all subscriptions (goroutine per subscription)
6. Log results; don't fail the job if one notification channel fails

Enqueue from handler:
```go
_, err = river.InsertTx(ctx, tx, ReminderArgs{TodoID: id}, &river.InsertOpts{
    ScheduledAt: reminderAt,
})
```

Use a transaction when inserting the River job alongside the todo update — both commit or both roll back.

To cancel a job when `reminder_at` is updated/cleared: store the River `job_id` in a `reminder_job_id` column on `todos` (add in a migration), then call `riverClient.JobCancel(ctx, jobID)` before enqueueing the new one.

## Frontend structure

### Auth flow

- Store `access_token` in memory (Zustand) and `refresh_token` in `localStorage`.
- On app load, if `refresh_token` exists, call `/auth/refresh` silently to get a new access token.
- `api/client.ts` — axios or native `fetch` wrapper that injects `Authorization` header, and on 401 automatically calls refresh then retries once.
- Redirect to `/login` if refresh fails.

### TanStack Query keys

```typescript
// Todos
['todos', view, categoryId, sort, order]    // list
['todos', id]                               // single

// Categories
['categories']
```

Invalidate `['todos', ...]` after any mutation (create, update, delete, snooze, done, reopen).

### Views

Each route maps to a view query param:

| Route | `view` param | Notes |
|---|---|---|
| `/app/today` | `today` | Also fetches `?view=overdue` for the collapsed section |
| `/app/upcoming` | `upcoming` | |
| `/app/overdue` | `overdue` | Full overdue list |
| `/app/done` | `done` | |
| `/app/category/:id` | — | Uses `category_id` filter |

### Today view — overdue section

Fetch both `today` and `overdue` lists. Display:
1. Today's todos sorted by priority then deadline
2. Collapsible section at bottom: "X overdue" — expand to show overdue list inline
3. Each overdue item shows a "Snooze" action (date picker popover) and "Done" button

### TodoItem component

Display: checkbox (done), title, category badge (colored dot + name), deadline date, priority indicator (colored left border: gray/blue/amber/red), reminder bell icon if `reminder_at` is set.

Actions accessible on hover/focus: edit (open sheet), snooze, delete.

Overline strikethrough on done items.

### AddTodoSheet / EditTodoSheet

Right-side sheet (shadcn/ui `Sheet`). Fields:
- Title (required, autofocus)
- Description (optional textarea)
- Deadline (date picker, defaults to today)
- Category (select, with "New category" inline creation)
- Priority (segmented control: none / low / medium / high)
- Reminder (datetime picker, optional)

On submit: optimistic update via TanStack Query.

### SortControls

Small toolbar above the todo list:
- Sort by: Deadline | Priority | Created (toggle buttons)
- Order: Asc / Desc

State lives in URL search params so sorting is shareable and survives refresh.

### Service Worker (Web Push)

`src/lib/sw.ts` — on login/app load:
1. Check `Notification.permission`
2. If granted (or on first prompt accept), call `serviceWorkerRegistration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: VAPID_PUBLIC_KEY })`
3. POST the subscription to `/api/v1/push/subscribe`

Service worker file (`public/sw.js`):
```javascript
self.addEventListener('push', event => {
    const data = event.data?.json() ?? {};
    event.waitUntil(
        self.registration.showNotification(data.title ?? 'ADHDoIt', {
            body: data.body,
            icon: '/icon-192.png',
            data: { url: data.url ?? '/' }
        })
    );
});

self.addEventListener('notificationclick', event => {
    event.notification.close();
    event.waitUntil(clients.openWindow(event.notification.data.url));
});
```

## Implementation order

Implement in this sequence to always have a runnable state:

1. **Scaffold** — repo layout, `go mod init`, `npm create vite`, compose skeletons, `.env.example`
2. **Database** — migration files, run `docker compose up -d db`, verify schema, run `sqlc generate`

## Key constraints and conventions

- Every DB query is user-scoped — always filter by `user_id` from JWT context. Never trust a user-supplied user_id.
- UUID primary keys everywhere. Generate with `uuid.New()` in Go or `gen_random_uuid()` in SQL.
- Timezone handling: `deadline` is a `DATE` (not timestamptz) to avoid TZ confusion. The API derives "today" from the user's `timezone` field when filtering. Store `reminder_at` as `TIMESTAMPTZ`.
- `status` is never directly settable via `PATCH /todos/:id` — use the dedicated action endpoints (snooze, done, reopen) to ensure invariants (e.g. snooze_until always set when snoozed).
- Overdue is derived, never stored — `deadline < (CURRENT_DATE AT TIME ZONE user.timezone)::date AND status = 'active'`.
- Soft-delete is not used. Hard delete only. The "done" status is the archive.
- CORS: allow the frontend origin in development (`localhost:5173`). In production, Caddy handles routing and the API should only allow the same origin.
- All handler tests: write table-driven tests using `net/http/httptest`. At minimum test the happy path and the unauthenticated case for each endpoint.
- Frontend: use `cn()` from shadcn for all conditional classNames. No inline style except where Tailwind is insufficient.
- Keep the River worker and the HTTP server in the same binary but separate goroutines. Add a build tag or env flag if they need to be split later.
- Generate VAPID keys once and store securely: `npx web-push generate-vapid-keys`

## Bugs & gotchas discovered during development

### Go handler: always assign all request fields to params
When adding a new handler, validate that every field from the request struct is
assigned to the DB params struct. Missing `params.Title = req.Title` caused todos
to silently be created with empty titles despite passing validation.

### Auth: single refresh, gate UI behind isAuthReady
`__root.tsx` and `apiFetch`'s 401 handler both call the refresh endpoint with the
same token. Token rotation means one call invalidates the other → logout loop.
Fix: `isAuthReady` bool in auth store; `__root.tsx` renders a blank screen until
the one initial refresh resolves, preventing TanStack Query from firing early.
See `src/store/auth.ts` (isAuthReady) and `src/routes/__root.tsx`.

### react-hook-form: custom controls must use Controller
Fields set only via `setValue()` without a `register()`-attached DOM element come
through as `undefined` in `handleSubmit` data → zod validation fails silently,
`onSubmit` is never called, no error shown to user. Always use `Controller` for
non-native inputs (segmented controls, custom pickers, etc.).

### TypeScript path aliases: paths without baseUrl
With `moduleResolution: "bundler"`, use `paths` alone — `baseUrl` is deprecated
(TS 6.0+) and causes build errors. `tsc` resolves `@/*` via `paths`; Vite resolves
it via `resolve.alias` in `vite.config.ts`. Both must be configured.

### Tailwind v4 migration
- PostCSS plugin moved: install `@tailwindcss/postcss`, use it instead of `tailwindcss` in postcss.config
- Replace `@tailwind base/components/utilities` with `@import "tailwindcss"`
- Use `@config "../tailwind.config.ts"` in CSS to keep v3-style config
- `autoprefixer` no longer needed in PostCSS chain (v4 built-in)

### Docker CI Dockerfiles
Dockerfiles live at `backend/Dockerfile` and `frontend/Dockerfile`.
Build context for each is their own subdirectory (`./backend`, `./frontend`).
The CI workflow uses `context: ./${{ matrix.component }}` and
`file: ./${{ matrix.component }}/Dockerfile`.
